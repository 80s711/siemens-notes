# 本地规则

本文件作用于当前目录 `E:\西门子嵌入式\西门子笔记` 及其子目录。

## 优先规则

- 在当前目录或其子目录开始任何新任务前，先读取本地 `AGENTS.md`，再执行后续操作。

## 默认约定

- 默认使用中文交流与写作。
- 该目录主要用于维护技术学习笔记，优先创建和编辑 Markdown 文件。
- 除非明确要求，不重命名、不删除已有文件。
- 修改现有笔记时，尽量保持原有结构和表述风格，只做增量调整。
- 当用户在本目录新增个人笔记后，后续补充和扩展必须以用户原有思路为主线，顺着已有结构、术语和讲解方式继续写。
- 若用户原文存在明显的推导顺序或学习路径，补充内容时应保持该顺序，不随意改成另一套组织方式。

## 写作规则

- 新内容优先使用清晰的 Markdown 结构，包括标题、列表、表格和代码块。
- 课程型或分讲笔记，开头优先补一小段总览，简要说明“这一讲主要学什么、解决什么问题、重点在哪里”，便于快速进入内容。
- 涉及代码、命令或配置时，提供可直接复制使用的示例。
- 涉及嵌入式、单片机、蓝桥杯、C 语言相关内容时，优先给出结论，再补充必要解释。
- 笔记内容以准确、简洁、可复习为目标，避免空泛描述。
- 语言表达尽量简单通俗，优先使用便于理解的说法，不刻意堆砌术语。
- 在不打断原有结构的前提下，可以加入简短的小例子、小场景或小对比，帮助理解，让笔记更生动。

## 编辑规则

- 新建笔记时，文件名应简洁并能直接体现主题。
- 若补充已有笔记，优先追加到对应章节，不随意拆散原内容。
- 若信息存在不确定性，应明确标注“待确认”。
- 对用户笔记做扩展时，优先补足定义、例子、易错点、对比说明和代码示例，不脱离原文主题单独发挥。
- 若在用户笔记中发现概念性、事实性、代码性或表述性错误，先明确指出问题及原因，等待用户确认后再修改原文。
- 在用户未明确同意前，可以给出修正建议，但不直接改动对应笔记内容。
- 只有当用户明确说明要修改某一篇笔记时，才能对该笔记文件执行编辑、补充、重组或修订操作。
- 若用户仅要求检查、阅读、分析或指出问题，则只输出结论与建议，不直接修改文件。
- 后续可根据用户的提问与回答过程，整理出适合沉淀进笔记的内容，如补充说明、例子、易错点和总结。
- 默认对用户提出的每一个问题都进行整理，提炼成便于复习、便于后续写入笔记的内容，而不是只做一次性回答。
- 以上整理内容只有在用户明确指定要修改哪篇笔记时，才能正式写入对应文件；未指定前只保留为建议或草稿思路。
- 笔记中使用的图片不要放在 `C` 盘或其他系统临时目录，统一存放在当前笔记目录下的 `imagine` 文件夹中。
- Markdown 中引用图片时，优先使用相对路径，如 `imagine/xxx.png`。
- 若发现笔记中的图片仍引用 Typora 临时目录、系统目录或其他不稳定路径，应优先转存到当前目录下的 `imagine` 文件夹，再更新为相对路径引用。
- 后续新增或整理笔记图片时，默认按“本地保存 + 相对路径引用”的方式处理，保证笔记迁移、备份和换电脑后仍可正常显示。
- 当前笔记目录已绑定 GitHub 公开仓库 `https://github.com/80s711/siemens-notes`，后续如需生成网页可访问的图片链接，默认基于该仓库处理。
- 需要给图片生成网页外链时，默认使用 GitHub Raw 链接格式：`https://raw.githubusercontent.com/80s711/siemens-notes/main/相对路径`，其中“相对路径”以当前仓库根目录为基准。
- 后续新增用于网页引用的图片时，仍优先保存到当前目录下的 `imagine` 文件夹，再按仓库内相对路径生成外链，不单独放到其他图床或临时目录。
- 若用户需要“可直接粘贴到网页或平台”的图片链接，默认同时给出图片外链和可直接复制的 Markdown/HTML 示例。
- 若用户未明确要求修改某篇笔记正文，则不要把现有 Markdown 中的相对路径批量改成 GitHub 外链；默认保留笔记内相对路径写法，外链单独输出供用户使用。
- 当新增图片并需要外链生效时，默认提醒或协助执行 `git add`、`git commit`、`git push`，确保 GitHub 上已有对应文件后再提供正式外链。

## 本地工具路径

- Typora 可执行文件路径固定为 `D:\typora\Typora\Typora.exe`。
- 如需从终端打开当前 Markdown 文件，优先复用以上路径，不重复搜索安装位置。
